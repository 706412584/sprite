## 关键技术决策

### 1. 数据存储架构

**决策**: 使用 CloudDataWrapper 封装 CloudData API

**理由**:
- 提供统一的数据访问接口
- 简化从 iApp Database 的迁移
- 集中处理错误和日志
- 支持分类存储（通过键名前缀）

### 2. 管理器架构

**决策**: 将 iApp 单例模式转换为 WasiCore IGameClass + 静态方法

**理由**:
- 符合 WasiCore 框架规范
- 避免单例模式的复杂性
- 更好的性能（静态方法）
- 清晰的生命周期管理

### 3. 转换工具策略

**决策**: 创建半自动化转换工具

**理由**:
- 完全自动化转换不现实（Java代码块、复杂逻辑）
- 半自动化可以处理大部分简单转换
- 保留人工审查的空间
- 提高迁移效率

### 4. 增量迁移策略

**决策**: 按依赖关系分阶段迁移

**理由**:
- 降低风险
- 每个阶段都可以编译验证
- 便于问题定位
- 支持并行开发

## 风险和挑战

### 高风险项 🔴

1. **Java 代码块转换** (Database.myu 有 1600+ 行 Java 代码)
   - 需要深入理解 Android API
   - 需要找到 WasiCore 等价 API
   - 可能需要重新设计部分功能

2. **UI 布局复杂性** (96 个 XML 布局)
   - Android 布局 → WasiCore 流式布局差异大
   - 需要手动调整布局逻辑
   - 响应式适配工作量大

3. **依赖关系复杂** (55 个管理器相互依赖)
   - 可能存在循环依赖
   - 需要仔细分析依赖图
   - 可能需要重构架构

### 中风险项 🟡

1. **数据结构差异**
   - iApp 分类存储 vs WasiCore 键值存储
   - 需要设计数据迁移方案

2. **安全特性移除**
   - Root 检测、模拟器检测等需要移除
   - 需要评估对游戏的影响

3. **性能优化**
   - WebAssembly 环境的性能特性
   - 需要针对性优化

### 低风险项 🟢

1. **基础语法转换** (规则明确)
2. **日志系统转换** (直接替换)
3. **条件编译** (清晰的客户端/服务端分离)
