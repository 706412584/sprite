## 3. 为什么先做前端-only

这是当前最优路线，原因如下：

### 3.1 先验证“布局表达”是否成立

当前最大风险不是引擎能不能渲染，而是：

- 这套节点模型是否够用
- 属性系统是否清晰
- 移动端页面壳子是否合理
- 编辑体验是否高效

这些问题都可以用前端独立验证。

### 3.2 降低首期复杂度

如果一开始就做：

- 编辑器
- 预览器
- JSON 协议
- Lua 导出
- C# 导出
- 引擎端解释器

复杂度会过高，而且协议尚未稳定就会被运行时绑定反向牵制。

### 3.3 更符合现有参考的抽象方向

从 `game_djbtz-master` 可以看出，真正可复用的不是业务逻辑，而是：

- 页面壳子
- Panel 容器树
- 滚动区域
- 顶栏 / 底栏 / 内容区
- 弹窗壳子
- 通用按钮组

这些完全可以独立于运行时先抽象出来。

## 7. JSON 协议设计

## 7.1 顶层结构

建议结构：

```json
{
  "version": "0.1.0",
  "page": {
    "id": "home_main",
    "name": "首页",
    "shell": "mobile_app",
    "devicePreset": "phone_portrait"
  },
  "theme": "xianxia_dark",
  "root": {}
}
```

## 7.2 节点结构

每个节点统一：

```json
{
  "id": "hud_root",
  "type": "Panel",
  "props": {},
  "style": {},
  "bindings": {},
  "actions": {},
  "children": []
}
```

字段职责：

- `id`：稳定节点 id
- `type`：节点类型
- `props`：组件功能属性
- `style`：布局与视觉属性
- `bindings`：绑定信息
- `actions`：事件 key
- `children`：子节点

## 7.3 绑定设计

只保存 key，不保存运行时代码：

```json
{
  "bindings": {
    "text": "player.realmName",
    "value": "player.cultivationProgress",
    "visible": "mail.unreadCount > 0"
  }
}
```

## 7.4 行为设计

同样只保存 key：

```json
{
  "actions": {
    "onClick": "OpenMail",
    "onLongPress": "OpenItemContext"
  }
}
```

设计原则：

- 行为不写成前端闭包函数
- 后续由不同 exporter 翻译成对应运行时调用

## 16. Lua / UrhoX 导出目标

后续导出到 Lua 时，目标不是生成完整业务逻辑，而是生成结构化 UI 树。

目标形态应接近：

```lua
return UI.Panel {
    id = "root",
    width = "100%",
    height = "100%",
    flexDirection = "column",
    children = {
        UI.Panel {
            id = "hud",
        },
        UI.ScrollView {
            id = "content",
        }
    }
}
```

Lua exporter 规则：

- 优先导出布局树
- `actions.onClick` 导出为事件占位
- `bindings` 导出为注释或映射 stub
- 不直接猜业务模块

## 21. 详细实现方案

本节补充“可以直接开工”的实现细节，重点解决：

- 文档对象模型怎么定
- 编辑器内部状态怎么流动
- 预览器怎么渲染
- 模板怎么实例化
- JSON 怎么校验和升级
- exporter 到底导出什么，不导出什么

---

## 21.1 文档对象模型（Document Model）

布局器内部建议统一采用一份文档对象模型。

### 顶层对象

```ts
type LayoutDocument = {
  version: string
  meta: LayoutMeta
  page: LayoutPage
  theme: string
  root: LayoutNode
}
```

### 页面元信息

```ts
type LayoutMeta = {
  id: string
  name: string
  createdAt: string
  updatedAt: string
  author?: string
  tags?: string[]
}
```

```ts
type LayoutPage = {
  id: string
  name: string
  shell: PageShellType
  devicePreset: DevicePresetId
  description?: string
}
```

### 节点对象

```ts
type LayoutNode = {
  id: string
  type: NodeType
  props: Record<string, unknown>
  style: Record<string, unknown>
  bindings?: Record<string, string>
  actions?: Record<string, string>
  children?: LayoutNode[]
}
```

### 关键约束

1. `id` 必须稳定
2. `type` 决定允许出现哪些 `props/style`
3. `children` 是否允许，取决于节点类型
4. `bindings/actions` 只允许字符串 key，不允许函数

---

## 21.2 节点约束表

第一阶段建议给每种节点定义一张能力表。

### `Panel`

- 可有子节点：是
- 可滚动：否
- 可 flex：是
- 可背景：是
- 可边框：是

### `Text`

- 可有子节点：否
- 必填属性：`text` 或 `bindings.text`
- 支持样式：
  - `fontSize`
  - `fontWeight`
  - `color`
  - `textAlign`

### `Button`

- 可有子节点：否（第一阶段先限制）
- 必填属性：
  - `text`
  - 或 `icon`
- 支持行为：
  - `onClick`

### `ScrollView`

- 可有子节点：是
- 必填属性：
  - `scrollX`
  - `scrollY`

### `List`

- 可有子节点：否（第一阶段）
- 必填属性：
  - `itemTemplateId`
  - `bindings.items`

### `Grid`

- 可有子节点：否（第一阶段）
- 必填属性：
  - `columns`
  - `itemTemplateId`

### `Tabs`

- 可有子节点：是
- 子节点必须为：
  - `TabPage`

这样做的目的是：

- 避免协议太松
- 降低 exporter 复杂度
- 降低预览器分支判断数量

---

## 21.3 样式值规范

第一阶段必须统一样式值格式，不允许自由发挥。

### 尺寸

统一允许：

- 数字：表示逻辑像素
- `"100%"`
- `"auto"`

例如：

- `width: 320`
- `width: "100%"`
- `height: "auto"`

### 间距

统一允许两种：

- 单值：`8`
- 四向对象：

```json
{ "top": 8, "right": 12, "bottom": 8, "left": 12 }
```

适用于：

- `padding`
- `margin`

### 颜色

第一阶段统一使用：

- `#RRGGBB`
- `#RRGGBBAA`
- 主题 token

例如：

- `"#1F2937"`
- `"theme.panel"`

### 圆角

- 第一阶段统一数字，不做椭圆角

### 显示

- `visible: true | false`
- 复杂条件放到 `bindings.visible`

---

## 21.4 编辑器状态管理

编辑器内部状态建议拆为四类。

### 1. 文档状态

- 当前 `LayoutDocument`
- 页面列表
- 当前页面

### 2. 选择状态

- 当前选中节点 id
- 多选节点 id 列表（第二阶段）

### 3. 视图状态

- 当前设备预设
- 当前缩放比例
- 当前预览模式
- 左右栏折叠状态

### 4. 历史状态

- undo stack
- redo stack

建议使用命令式变更模型：

```ts
type EditorCommand =
  | AddNodeCommand
  | RemoveNodeCommand
  | MoveNodeCommand
  | UpdateNodePropCommand
  | UpdateNodeStyleCommand
  | DuplicateNodeCommand
```

好处：

- 方便撤销/重做
- 方便日志记录
- 方便后续协作编辑扩展

---

## 21.5 节点树编辑实现

第一阶段不要先做复杂拖拽，推荐这样落地：

### 左侧树操作

- 添加子节点
- 插入同级节点
- 上移
- 下移
- 删除
- 复制

### 中间预览区交互

- 点击选中节点
- hover 高亮节点边框
- 双击可跳到节点树

### 右侧属性面板交互

- 修改属性后立即更新文档
- 所有修改都走命令系统

这样就能形成一个闭环：

1. 选择节点
2. 修改属性
3. 实时预览
4. 可撤销

---

## 21.6 模板系统实现

模板不能只是“复制一份 JSON”，需要分两层。

### 1. 页面模板

例如：

- 首页模板
- 商店模板
- 背包模板

页面模板包含：

- page meta
- shell
- root 节点树

### 2. 组件模板

例如：

- HUD
- 顶部标题栏
- 底部 Tab
- 物品格子
- 属性行

组件模板只是一棵可插入节点树。

### 模板实例化规则

实例化时必须做：

1. 为所有节点重新生成 `id`
2. 保留原模板内部结构
3. 可选写入 `templateSource`

示例：

```ts
type TemplateSourceInfo = {
  templateId: string
  templateVersion: string
}
```

第一阶段建议：

- 模板实例化后和源模板解耦
- 不做“模板联动更新”

不然复杂度会上去很多。

---

## 21.7 预览渲染规则

预览器建议实现成：

```ts
function renderNode(node: LayoutNode): ReactNode
```

按 `type` 分派到不同 renderer。

### 渲染层结构

```text
PreviewRenderer
  -> DeviceFrame
  -> SafeAreaFrame
  -> PageShellRenderer
  -> NodeRenderer(root)
```

### 预览渲染约束

1. 预览器必须只认 schema
2. 不允许组件自己偷偷依赖外部业务对象
3. 所有节点必须能在“无真实数据”下渲染 placeholder

### 绑定预览策略

例如：

- `player.realmName` 没有真实值时
- 使用 mock 数据层提供默认值

建议增加：

```ts
type PreviewDataContext = Record<string, unknown>
```

这样模板和页面能用 mock 数据预览。

---

## 21.8 页面壳子（Shell）实现

页面壳子不应只是一个字符串，而应有明确渲染规则。

### `mobile_app`

结构固定：

- top slot
- content slot
- bottom slot
- overlay slot

### `modal_dialog`

结构固定：

- backdrop
- dialog container
- content slot
- footer slot

### `tab_page`

结构固定：

- tab header
- tab content

### 实现方式

推荐：

- shell 本身也是内建模板
- 但在预览器里由专门 renderer 控制结构

原因：

- 这样页面作者不用每次手拼整个 App 壳子
- 也便于不同运行时统一导出

---

## 21.9 属性面板实现

属性面板不应直接硬编码所有输入框，建议采用 schema 驱动。

### Inspector Schema

给每种节点类型定义 inspector 配置：

```ts
type InspectorField = {
  key: string
  label: string
  editor: "text" | "number" | "select" | "color" | "switch" | "spacing"
  section: "props" | "style" | "bindings" | "actions"
}
```

例如 `Panel` 会有：

- `flexDirection`
- `justifyContent`
- `alignItems`
- `gap`
- `padding`
- `margin`
- `background`

这样做的好处：

- 节点增加字段时不用重写整个属性面板
- 后续还能做自动校验提示

---

## 21.10 JSON 校验与版本迁移

这是后面最容易被忽略但最值钱的一层。

### 校验

建议做两层：

1. 基础结构校验
   - 是否有 `version`
   - 是否有 `root`
   - 节点字段是否合法

2. 节点语义校验
   - `Text` 不能有 children
   - `Grid` 必须有 columns
   - `Tabs` 不能直接挂 Text 等

### 版本迁移

建议从第一天就预留：

```ts
type DocumentMigrator = (oldDoc: unknown) => LayoutDocument
```

版本升级路径示例：

- `0.1.0 -> 0.2.0`
- `0.2.0 -> 0.3.0`

原则：

- 任何破坏性字段变更都必须配 migrator

---

## 21.11 数据绑定与行为边界

这里必须钉死，不然后面 exporter 会失控。

### 布局器负责

- 保存绑定 key
- 保存事件 key
- 保存条件显示表达式

### 布局器不负责

- 真实业务数据读取
- 真实事件执行逻辑
- 网络请求
- 游戏状态变更

例如：

- 可以写 `bindings.text = "player.name"`
- 可以写 `actions.onClick = "OpenMail"`
- 但不能写：
  - `onClick: () => sendCloudReq(...)`

---

## 21.12 Exporter 详细边界

### JSON Exporter

职责：

- 输出规范化文档
- 去掉编辑器内部临时状态

### Lua Exporter

职责：

- 把结构化节点转成 `UI.Panel/UI.Label/...`
- 保留稳定 id
- 输出 stub 注释：
  - bindings
  - actions

不负责：

- 自动接入业务模块
- 自动写 handler 逻辑

### C# Exporter

职责：

- 输出页面骨架
- 输出组件树构建代码
- 输出查找 id 帮助方法

不负责：

- 自动生成 Manager 逻辑
- 自动生成完整事件实现

---

## 21.13 首期不做什么

为了控范围，第一阶段明确不做：

1. 任意自由拖拽画布
2. 复杂时间轴动画
3. 富文本编辑器
4. 引擎内实时预览
5. 模板联动更新
6. 多人协作
7. 自动生成完整业务代码
8. 复杂公式表达式编辑器

这几项都很容易让项目失速。

## 24. 再次审查结论

基于当前所有参考对象，再审查一次，这条路线仍然成立。

### 审查结论 1

现有 Lua 参考已经证明：

- 移动端 UI 的主要共性非常强
- 完全值得抽象成通用容器树

### 审查结论 2

当前最大价值不是“直接导出运行时代码”，而是：

- 先统一布局表达
- 先提高做页面的速度

### 审查结论 3

前端-only 路线依然是最稳解：

- 成本最低
- 验证最快
- 风险最小

### 审查结论 4

最大的风险不是技术选型，而是范围失控。

所以必须坚持：

- 第一阶段只做结构化布局
- 不做自由画布
- 不做复杂运行时耦合
